01 — Executive Summary

What This Platform Does

AppSec AI Platform is an enterprise-grade web application security audit platform that combines a structured Python automation layer with a governed intelligence layer. It delivers consistent, evidence-based security assessments and produces professional audit reports tailored to three distinct audiences: security teams, leadership, and engineering.

The platform enforces written authorization before any audit activity begins, operates in passive review mode by default, and requires labeled evidence behind every finding before it enters the register. These are not documentation guidelines — they are code-level controls. The result is audit output that is defensible, consistent, and ready for compliance or management review.

Phase 1 is operational. Passive web application security assessment, evidence collection, findings normalization, and HTML report generation are fully implemented. Active testing capabilities are architecturally present but gated behind explicit authorization. Engineering AppSec extensions (SAST, IaC, cloud) are on the development roadmap.

02 — Business Problem

The Problem This Platform Solves

Security reviews conducted without a structured platform produce inconsistent results, lack defensible evidence trails, and fail to communicate risk clearly to the right audiences.

ProblemImpact
Ad hoc, inconsistent assessments Audit quality varies by analyst; results cannot be meaningfully compared across time or teams
No systematic evidence collection Findings are difficult to defend, challenge, or track across engagement cycles
No defined remediation timelines Issues age without accountability; risk is not managed to closure
Single-audience reporting Technical detail overwhelms leadership; non-technical summaries miss critical context for engineering
Authorization not enforced Testing may proceed without written approval — a legal, compliance, and reputational risk

03 — Solution Overview

How the Platform Is Structured

The platform operates as two complementary layers that work together in every audit engagement.

LayerLocationResponsibility
Python Automation Layer src/ · scripts/ CLI-driven execution, 13 audit tools, 5 parsers, evidence and findings writers, report generators
Governance and Intelligence Layer .claude/ Authorization rules, scope enforcement, evidence quality standards, severity criteria, remediation SLAs, domain-specific review skills and templates

Every audit engagement starts by reading context files that define the target, authorization status, scope boundaries, and assumptions. The authorization gate is verified at the code level before any tool executes. All findings are normalized to a consistent structure and accumulate in an append-only findings register that feeds all report generation.

Design principle: The platform cannot produce a finding without a documented evidence basis, cannot begin audit activity without confirmed written authorization, and cannot test targets outside the defined scope. These constraints are enforced at runtime, not just documented in policy.

05 — End-to-End Workflow

From Setup to Publication

The platform follows a strict seven-stage engagement workflow. Authorization is verified before execution begins; evidence is captured automatically; reports are generated from the findings register at any point in the engagement.

  1. 1

    Platform Setup

    Install Python dependencies, configure .env with the target URL and auditor details, provision the virtual environment.

    pip install -r requirements.txt
  2. 2

    Authorization and Context

    Populate .claude/context/audit-context.md with CONFIRMED authorization status, the authorizing party, and a reference to the authorization document. Define in-scope and out-of-scope targets in scope.md.

    Code-level enforcement — execution halts without CONFIRMED status
  3. 3

    Audit Execution

    Run the CLI with the selected domain or full workflow. The authorization gate is checked first. Passive audit tools execute in sequence across all in-scope targets.

    python scripts/run_audit.py audit full
  4. 4

    Evidence Capture

    Each audit tool automatically writes EVID- labeled evidence files to evidence/raw/. The auditor reviews and promotes evidence through the three-stage pipeline before report inclusion.

    evidence/raw/ → reviewed/ → summarized/
  5. 5

    Findings Registration

    Normalized FIND-NNN entries are appended to the findings register with severity, confidence, domain, recommendation, and evidence reference. The register is the single source of truth for the engagement.

    audit-runs/active/findings-register.md
  6. 6

    Report Generation

    Three report types are generated from the findings register in both Markdown and self-contained HTML formats. HTML versions include severity distribution charts.

    python scripts/run_audit.py report technical | executive | remediation
  7. 7

    Site Publication

    The HTML site builder assembles all reports into a GitHub Pages portal for stakeholder distribution and management review.

    python scripts/build_reports_site.py

06 — Governance and Safety Controls

How the Platform Stays Safe and Accountable

Six rule sets govern every engagement. These are enforced at both the documentation and code levels — not aspirational policies.

01
Authorization Enforcement

All audit workflows verify .claude/context/audit-context.md for a CONFIRMED authorization status at runtime. No testing activity of any kind begins without written authorization from the named authorizing party.

02
Passive-by-Default Mode

The default operating mode is passive review — reading and analyzing observable security properties without submitting test payloads. Active testing requires explicit written authorization and is architecturally gated behind an authorization override. Production environments are protected by this default.

03
Scope Enforcement

Out-of-scope targets defined in scope.md are blocked at the workflow level. Out-of-scope conditions observed passively are labeled [OUT OF SCOPE — NOT TESTED] and documented without further investigation. Scope expansion requires written authorization and a logged change entry.

04
Mandatory Stop Conditions

The platform must halt all audit activity immediately if unexpected PII, payment card data, plaintext credentials, private encryption keys, or indicators of active compromise are discovered. The auditor is alerted and must seek explicit guidance before continuing.

05
Evidence Quality Rules

Every finding must reference at least one EVID- labeled piece of evidence before entering the findings register. Speculative findings are prohibited in formal reports. Evidence without a date and attribution may not support a finding. Sensitive data discovered in evidence must be redacted before report inclusion.

06
Severity and Remediation Standards

Five severity levels — Critical, High, Medium, Low, Info — are aligned to CVSS v3 guidance. Severity overrides require documented justification. Remediation SLAs are defined and tracked: Critical 24 hours, High 7 days, Medium 30 days, Low 90 days. Risk acceptance requires management sign-off and an expiry date of no more than six months.

10 — Summary

Management Takeaways

Key points for leadership and internal stakeholders evaluating this platform.

🔒

Authorized and Auditable by Design

Every audit run is gated behind confirmed written authorization. Every finding references labeled evidence. The platform cannot produce unsupported findings or proceed without explicit approval.

🔁

Consistent and Repeatable

Standardized tooling and governance rules eliminate analyst-to-analyst variability. The same checks run the same way on every engagement, enabling meaningful comparison across time.

📊

Multi-Audience Reporting

One findings register produces three distinct reports: a technical deep-dive for the security team, a plain-language executive summary for leadership, and a prioritized remediation plan for engineering.

🛡

Safe by Default

Passive review is the default mode. The platform will not perform active security testing unless explicitly authorized in writing. Production environments are protected by this architectural default.

⏱

SLA-Driven Remediation

Every finding carries a specific, actionable recommendation and a defined resolution timeline. Critical issues require a response within 24 hours. Risk acceptance requires management sign-off and a fixed expiry.

🚀

Phase 1 Operational

Passive web security assessment and professional reporting are fully implemented and production-ready. Active testing and engineering AppSec extensions (SAST, IaC, cloud) are the Phase 2 roadmap.
